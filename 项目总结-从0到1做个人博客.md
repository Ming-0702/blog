### 3.2 后端分层架构（MVC 模式）

```

│   │   └── v1/        #   版本 v1 的路由
```

> **对应文档中的 MVC 三层**：
> - Controller = `api/v1/` 目录 —— 解析请求，返回响应
> - Service = `services/` 目录（当前简单逻辑直接写在 API 层，预留供复杂逻辑使用）
> - DAO = `models/` + SQLAlchemy session —— ORM 与数据库交互

### 4.2 核心表设计

#### `users` 表 — 用户
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增主键 |
| username | VARCHAR(50) UNIQUE | 用户名 |
| email | VARCHAR(100) UNIQUE | 邮箱 |
| hashed_password | VARCHAR(255) | bcrypt 哈希密码 |
| nickname | VARCHAR(50) | 昵称 |
| avatar_url | VARCHAR(500) | 头像 URL |
| bio | VARCHAR(500) | 个人简介 |
| created_at | TIMESTAMP | 注册时间 |

#### `posts` 表 — 文章
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增主键 |
| title | VARCHAR(200) | 标题 |
| content | TEXT | 正文 |
| summary | VARCHAR(500) | 摘要 |
| status | VARCHAR(20) | draft / published |
| author_id | INTEGER FK → users.id | 作者 |
| view_count | INTEGER | 阅读数 |
| like_count | INTEGER | 点赞数 |
| comment_count | INTEGER | 评论数 |

#### `comments` 表 — 评论
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增主键 |
| content | TEXT | 评论内容 |
| post_id | INTEGER FK → posts.id | 所属文章 |
| author_id | INTEGER FK → users.id | 评论者 |
| parent_id | INTEGER FK → comments.id | **楼中楼（自引用）** |

#### `likes` 表 — 点赞（多态设计）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增主键 |
| user_id | INTEGER FK | 点赞用户 |
| target_type | VARCHAR(20) | "post" 或 "comment" |
| target_id | INTEGER | 目标对象 ID |
| UNIQUE(user_id, target_type, target_id) | | 防止重复点赞 |

> **设计亮点**：点赞表使用 `target_type` + `target_id` 的**多态关联**设计，一张表同时支持文章和评论点赞，未来要加新的点赞对象类型也只需扩展枚举值。

## 5. 后端实现（FastAPI）

### 5.1 项目配置 (`core/config.py`)

使用 `pydantic-settings` 管理配置，支持 `.env` 文件和环境变量：

```python
class Settings(BaseSettings):
    DATABASE_URL: str = "postgresql+asyncpg://postgres:postgres@localhost:5432/blog"
    SECRET_KEY: str = "your-secret-key-change-in-production"
    ALGORITHM: str = "HS256"
    ACCESS_TOKEN_EXPIRE_MINUTES: int = 60 * 24 * 7  # 7天
    CORS_ORIGINS: list[str] = ["http://localhost:5173"]
```

### 5.2 异步数据库 (`core/database.py`)

使用 SQLAlchemy 2.0 异步引擎：

```python
engine = create_async_engine(settings.DATABASE_URL, echo=settings.DEBUG)
async_session = async_sessionmaker(engine, class_=AsyncSession, expire_on_commit=False)
```

FastAPI 通过依赖注入获取数据库会话：

```python
async def get_db():
    async with async_session() as session:
        yield session
```

### 5.3 JWT 认证 (`core/security.py`)

- **密码存储**：使用 `passlib` + `bcrypt` 哈希，不存储明文
- **令牌签发**：使用 `python-jose` 创建 JWT，payload 包含 `user_id` 和 `exp`
- **令牌验证**：每个受保护接口通过 `get_current_user` 依赖自动解析

```python
# 注册时
hashed_password = hash_password(password)

# 登录时
if not verify_password(plain_password, hashed_password):
    return fail("用户名或密码错误")

# 生成令牌
access_token = create_access_token({"user_id": user.id})
```

### 5.4 认证 API (`api/v1/auth.py`)

| 端点 | 方法 | 说明 | 鉴权 |
|------|------|------|------|
| `/api/v1/auth/register` | POST | 用户注册 | 否 |
| `/api/v1/auth/login` | POST | 用户登录 | 否 |
| `/api/v1/auth/me` | GET | 获取当前用户 | 是 |

**统一响应格式**（遵循文档规范）：
```json
{
    "code": 0,
    "msg": "success",
    "data": { ... }
}
```

### 5.5 文章 API (`api/v1/posts.py`)

| 端点 | 方法 | 说明 | 鉴权 |
|------|------|------|------|
| `/api/v1/posts` | GET | 文章列表（分页） | 否 |
| `/api/v1/posts/{id}` | GET | 文章详情（+1 阅读） | 否 |
| `/api/v1/posts` | POST | 创建文章 | 是 |
| `/api/v1/posts/{id}` | PUT | 更新文章 | 是（作者） |
| `/api/v1/posts/{id}` | DELETE | 删除文章 | 是（作者） |

### 5.6 评论 API — 含楼中楼 (`api/v1/comments.py`)

楼中楼的实现通过 `parent_id` 自引用完成：

```python
class Comment(Base):
    parent_id = Column(Integer, ForeignKey("comments.id"), nullable=True)
    replies = relationship("Comment", back_populates="parent", remote_side=[id])
```

获取评论时：先取顶级评论（`parent_id IS NULL`），再取所有回复（`parent_id IS NOT NULL`），在 Python 中组装。

### 5.7 点赞 API (`api/v1/likes.py`)

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/v1/likes` | POST | **切换**点赞状态（已赞则取消，未赞则点赞） |
| `/api/v1/likes/status` | GET | 查询当前用户是否已赞 |

> **设计亮点**：点赞/取消用一个端点处理，前端不需要维护点赞状态切换逻辑。

## 6. 前端实现（React + Ant Design）

### 6.1 认证流程

```
注册/登录 ──▶ 后端返回 JWT ──▶ 存入 localStorage
                                      │
                               Axios 拦截器自动附加
                                      │
                              请求头: Authorization: Bearer <token>
                                      │
                              401 响应 ──▶ 自动跳转登录页
```

### 6.2 API 请求层 (`api/client.js`)

使用 Axios 实例，通过**请求拦截器**自动附加 JWT，通过**响应拦截器**统一处理 401：

```javascript
// 请求拦截器
api.interceptors.request.use((config) => {
  const token = localStorage.getItem('token');
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// 响应拦截器
api.interceptors.response.use(
  (response) => response.data,
  (error) => {
    if (error.response?.status === 401) {
      localStorage.removeItem('token');
      window.location.href = '/login';
    }
    return Promise.reject(error.response?.data);
  }
);
```

### 6.3 状态管理 (`contexts/AuthContext.jsx`)

使用 React Context 管理全局认证状态，无需引入 Redux 等重型状态管理：

```javascript
<AuthProvider>  // 包裹整个应用
  {children}    // 所有子组件通过 useAuth() 获取 user/loading/login/logout
</AuthProvider>
```

### 6.4 页面路由

```
/              → Home.jsx        首页（展示最新文章）
/login         → Login.jsx       登录
/register      → Register.jsx    注册
/posts         → PostList.jsx    文章列表（分页）
/posts/new     → CreatePost.jsx  写文章（需登录）
/posts/:id     → PostDetail.jsx  文章详情 + 评论 + 点赞
/posts/:id/edit→ EditPost.jsx    编辑文章（需登录）
```

### 6.5 关键交互设计

**点赞**：使用 Ant Design 的 `HeartOutlined` / `HeartFilled` 图标切换，点击即切换状态，无需页面刷新。

**楼中楼评论**：在 `PostDetail.jsx` 中，评论列表展开嵌套的 `replies` 数组，区分顶级评论和回复，支持"回复"按钮设置 `replyTo` 状态。

**分页**：`PostList.jsx` 使用 Ant Design 的 `Pagination` 组件，切换页码时重新请求 API。

### 7.2 前端 Dockerfile（多阶段构建）

```dockerfile
# 第一阶段：构建
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# 第二阶段：运行时（Nginx）
FROM nginx:alpine
COPY --from=build /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
```

> **为什么用 Nginx？** 前端是 SPA（单页应用），需要 Nginx 做：① 静态文件托管 ② API 反向代理 ③ SPA 路由回退（`try_files $uri /index.html`）

### 7.3 Nginx 配置关键点

```nginx
# SPA 路由支持：所有路径回退到 index.html
location / {
    try_files $uri $uri/ /index.html;
}

# API 反向代理到后端
location /api/ {
    proxy_pass http://backend:8000;
}
```

### 7.4 开发时 Vite 代理

开发环境中，通过 Vite 的 proxy 配置解决跨域：

```js
// vite.config.js
server: {
    proxy: {
        '/api': {
            target: 'http://localhost:8000',
            changeOrigin: true,
        },
    },
}
```

这样前端 API 路径统一写成相对路径 `/api/v1/...`，开发时 Vite 代理到后端，生产时 Nginx 代理到后端。

## 8. 进阶功能：MCP Server

### 8.1 什么是 MCP？

MCP（Model Context Protocol）是 Anthropic 开放的协议，它让 AI Agent（如 Claude）能够以标准化的方式与外部工具和数据源交互。

简而言之：**MCP Server 是 AI Agent 的"USB 接口"**——插上之后，Agent 就能直接操作你的系统。

### 8.2 实现的 9 个 MCP 工具

| 工具名称 | 功能 | 需要鉴权 |
|----------|------|----------|
| `list_posts` | 查看文章列表（分页） | ❌ |
| `get_post` | 查看文章详情 | ❌ |
| `search_posts` | 关键词搜索文章 | ❌ |
| `create_post` | 发布新文章 | ✅（需用户名+密码） |
| `update_post` | 编辑文章 | ✅（需作者鉴权） |
| `delete_post` | 删除文章 | ✅（需作者鉴权） |
| `list_comments` | 查看文章评论 | ❌ |
| `blog_stats` | 博客统计概览 | ❌ |
| `get_user_info` | 查询用户信息 | ❌ |

### 8.3 鉴权设计思路

写操作（增/删/改）需要鉴权，这里直接复用了已有的用户密码体系，要求用户传入 `username` + `password`：

```python
@mcp.tool(description="发布一篇新文章（需要提供用户名和密码进行鉴权）")
async def create_post(username: str, password: str, title: str, content: str, ...):
    # 1. 验证用户身份
    user = await authenticate(username, password)
    # 2. 执行操作
    post = Post(title=title, content=content, author_id=user.id)
```

### 8.4 两种传输模式

```bash
# 模式1：stdio（给 Claude Desktop / Claude Code 用）
python -m app.mcp_server --transport stdio

# 模式2：SSE（给其他 MCP 客户端用，支持 HTTP）
python -m app.mcp_server --transport sse --port 8100
```

### 8.5 技术实现

使用 `mcp[cli]` 的 `FastMCP` 框架：

```python
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("MyBlog MCP Server")

@mcp.tool(description="查看文章列表")
async def list_posts(page: int = 1, page_size: int = 20) -> str:
    # ... 查询数据库，返回格式化文本
```

每个工具就是一个 `async def` 函数 + `@mcp.tool` 装饰器，框架负责处理 MCP 协议的底层通信。

## 9. 答辩常问问题

> 以下是 ref 文档和实际开发中常见的问题，提前准备好答案：

### Q1: 为什么选用 React + FastAPI 这个技术栈？

**答**：文档推荐了 React + Go 和 React + Python FastAPI 两条路线。我选择 FastAPI 是因为：
1. Python 是我熟悉的语言，开发效率高
2. FastAPI 原生支持异步，性能接近 Go 的水平
3. 自动生成 OpenAPI 文档，开发和测试都很方便
4. Python 生态对 AI/ML 的支持更好，方便后续扩展 AI 功能

### Q2: 讲讲你的项目结构，每个目录大概放了哪些代码？

**答**：后端采用 MVC 模式：
- `api/v1/`：Controller 层，放路由处理函数，每个模块一个文件（auth/posts/comments/likes）
- `core/`：基础设施层，包括配置加载、数据库连接、JWT/密码工具
- `models/`：SQLAlchemy ORM 模型，对应数据库表结构
- `schemas/`：Pydantic 数据模型，定义请求和响应的数据格式
- `utils/`：工具函数，比如统一响应格式

前端按功能模块分：
- `api/`：API 请求封装
- `pages/`：页面组件
- `components/`：通用组件（导航栏）
- `contexts/`：全局状态（认证信息）

### Q3: vibecoding 的体验怎么样？

**答**：Vibe Coding 能极大提升开发效率，尤其适合快速搭建项目骨架。但我的体会是，**理解生成的代码比生成代码本身更重要**。

比如 MCP Server 的实现，AI 可以快速生成 9 个工具函数的框架，但数据库连接方式、鉴权策略、错误处理这些关键设计，还是需要我理解业务逻辑后做决策。

文档中也提到：**至少目前在某些领域由于相关语料的匮乏等原因，我们仍需自己动手。** 基础知识的积累在 debug 和调优时尤为重要。

### Q4: 为什么选择 MCP Server 作为进阶功能？

**答**：MCP Server 是文档重点推荐的方向，我认为它最有"实用性"——它让博客不再只是一个展示网站，而是可以被 AI Agent 管理。比如我可以直接用自然语言让 Claude Code 帮我发文章、查数据，真正把博客变成了一个"可编程"的产品。

同时，MCP 是 Anthropic 的新协议，学习它对理解 AI 工具的发展方向也很有帮助。

### Q5: 有哪些独特的感受或收获？

**答**：最大的收获是**"系统性思维"**——以前写代码可能只关注功能实现，但这个项目让我体会到了从需求分析 → 数据库设计 → API 设计 → 前端开发 → 部署运维的全流程。

特别是楼中楼评论和点赞表的多态设计，这些"看起来简单"的功能背后其实有挺多设计考量。比如点赞表用 `target_type` + `target_id` 的多态设计，而不是给每种点赞对象单独建表，这种"用设计换扩展性"的思路让我印象很深。

### Q6: 部署流程是怎样的？

**答**：使用 Docker Compose 一键部署。涉及三个容器：
1. **PostgreSQL**：数据持久化
2. **FastAPI 后端**：Gunicorn + Uvicorn 运行
3. **Nginx + React 前端**：Nginx 托管静态文件并反向代理 API

前端采用多阶段构建，先 `npm run build` 生成静态文件，再用 Nginx 提供服务。Nginx 同时承担 SPA 路由回退和 API 代理的功能。

### Q7: JWT 认证的流程是怎样的？

**答**：用户登录后，后端验证密码，签发 JWT（含 user_id 和过期时间）。前端将 token 存入 localStorage，之后每次请求通过 Axios 拦截器自动附加 `Authorization: Bearer <token>` 头。后端通过 `get_current_user` 依赖注入验证 token 并获取用户信息。如果 token 过期或无效，返回 401，前端拦截器自动跳转到登录页。

### 10.1 关键技术点总结

| 知识点 | 学到了什么 |

| **React 19** | 函数组件、Hooks、Context、路由 |
| **Ant Design** | Form、Card、List、Pagination、Dropdown |
| **Axios** | 请求/响应拦截器、JWT 自动附加、401 统一处理 |
| **Nginx** | SPA 配置、反向代理、多阶段构建 |
| **Docker** | Compose 编排、多阶段构建、健康检查 |
| **MCP** | FastMCP 框架、Tool 定义、stdio/SSE 传输 |
| **楼中楼设计** | 自引用外键实现嵌套评论 |
| **多态关联** | target_type + target_id 实现通用点赞 |

### 10.2 设计决策复盘

| 决策 | 为什么这样做 | 有没有更好的方案 |
|------|------------|----------------|
| 点赞表用多态关联 | 一张表支持文章和评论点赞，扩展性好 | 也可以用 JSON 单字段存储目标信息，但查询效率低 |
| 评论取两级分开查询 | 顶级评论 + 全部回复，Python 中组装 | 可以用递归 CTE 一次查询，但复杂度高且不大适合这个场景 |
| JWT 存储 user_id 而不是 username | user_id 是稳定标识符，改名不影响 | 两者均可 |
| 前后端通过代理通信 | 开发时 Vite proxy，生产时 Nginx proxy，无需配置 CORS | 也可以前后端直接对接，但需要配置 CORS 头 |

### 10.3 后续可以做的改进

2. **语义搜索**：接入向量数据库 / 嵌入模型，实现 "搜索"优化"也能搜到"索引"内容
3. **头像上传**：接入云存储（阿里云OSS / AWS S3）
4. **第三方登录**：GitHub OAuth
5. **单元测试**：给后端 API 写 pytest 测试
6. **WebSocket**：实现在线人数统计、实时通知
7. **数据仪表盘**：用户行为分析和博客数据可视化
